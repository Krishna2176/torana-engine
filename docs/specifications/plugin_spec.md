# Plugin Specification

**Version:** 0.1  
**Status:** Active  
**Last Updated:** July 2026

---

# Purpose

This document defines the implementation contract for all TORANA Plugins.

Every Plugin must implement the abstract Plugin base class.

The purpose of this specification is to ensure that all Plugins expose a consistent interface, allowing the Engine to execute analyses without knowledge of their internal implementation.

---

# Design Goals

The Plugin contract is designed to be:

- Consistent
- Predictable
- Extensible
- Testable
- Independent of specific analyses

Every Plugin should be interchangeable from the perspective of the Engine.

---

# Plugin Contract

Every Plugin must inherit from the abstract `Plugin` base class.

The Plugin contract consists of:

- Identity
- Metadata
- Configuration
- Dataset requirements
- Workflow definition
- Output declaration
- Validation

---

# Required Properties

Every Plugin must implement the following properties.

## Identity

- `id`
- `name`
- `version`
- `description`

These properties uniquely identify the Plugin.

---

## Metadata

```text
metadata
```

Additional descriptive information about the Plugin.

Typical values include:

- author
- license
- documentation
- tags

---

## Configuration

```text
configuration_schema
```

Describes the configuration accepted by the Plugin.

The Engine uses this schema for validation before execution begins.

---

## Dataset Requirements

Every Plugin declares its required datasets.

```text
required_datasets
```

Datasets that must exist before execution.

```text
optional_datasets
```

Datasets that improve the analysis but are not mandatory.

---

## Workflow

```text
workflow
```

Returns the Workflow describing the execution pipeline for the analysis.

The Workflow is responsible for task ordering.

---

## Outputs

```text
outputs
```

Declares the products generated by the Plugin.

Typical outputs include:

- rasters
- vectors
- reports
- statistics
- tables

---

# Required Methods

## validate()

Every Plugin must implement:

```python
validate() -> bool
```

The method should verify:

- configuration validity
- required parameters
- dataset availability (where appropriate)

The Engine invokes validation before execution.

---

# Responsibilities

A Plugin **must**:

- describe the analysis
- define configuration
- declare required datasets
- define a workflow
- declare outputs
- validate configuration

A Plugin **must not**:

- schedule tasks
- execute workflows
- manage jobs
- manage datasets
- perform visualization
- interact directly with the Engine

---

# Registration

Plugins are registered with the Plugin Registry.

The Registry is responsible for:

- uniqueness
- discovery
- lookup

Plugins should never register themselves.

---

# Error Handling

Plugins should raise meaningful exceptions when validation fails.

Validation errors should be deterministic and informative.

---

# Example Structure

A typical Plugin consists of:

```text
MyPlugin

├── Identity
├── Metadata
├── Configuration
├── Dataset Requirements
├── Workflow
├── Outputs
└── Validation
```

---

# Compatibility

Plugins should remain compatible with the current Plugin contract.

Breaking changes to the contract require a new major version of the Engine.
