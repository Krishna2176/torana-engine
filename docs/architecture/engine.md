# TORANA Engine

## Overview

The TORANA Engine is the execution core of the TORANA platform. It orchestrates the complete lifecycle of a geospatial analysis, from a user request to the generation of final outputs.

The engine is designed to be generic and independent of any specific analysis domain. It does not contain algorithms for Urban Heat Island, Flood Risk, Walkability, Solar Potential, or any other spatial analysis. Instead, it coordinates analysis plugins that encapsulate domain-specific knowledge.

The engine is responsible for execution, orchestration, resource management, workflow coordination, logging, caching, and fault tolerance.

---

# Vision

The TORANA Engine aims to provide a scalable, reproducible, and extensible framework for automated geospatial computation.

Every analysis should execute through the same engine regardless of its complexity or scientific domain.

The engine should remain stable even as new analysis plugins are introduced.

---

# Design Philosophy

TORANA follows a **plugin-driven execution model**.

The Engine knows **how to execute**.

Plugins know **what to execute**.

This separation ensures that the engine never requires modification when new analyses are added.

Every execution follows a common lifecycle while allowing each plugin to define its own datasets, algorithms, and outputs.

---

# Guiding Principles

The engine is designed around the following principles:

* Generic execution architecture
* Plugin independence

* Reproducible workflows

# Core Concepts

## Engine

The Engine is the long-lived execution coordinator of TORANA.

It is responsible for:

* Managing execution services
* Loading plugins
* Creating jobs
* Coordinating workflows
* Scheduling tasks
* Managing execution resources
* Providing logging and caching services

The Engine never performs analysis-specific computations.

---

## Job

A Job represents one execution of the TORANA Engine.

Every time a user requests an analysis, the Engine creates a new Job.

The Job becomes the root object for all execution-specific state.

Each Job owns:

* User request
* Configuration
* Selected plugin
* Workflow instance
* Task states
* Dataset references
* Intermediate results
* Outputs
* Execution logs
* Metadata
* Execution status

Jobs are independent of one another and may execute concurrently.

---

## Plugin

A Plugin implements one geospatial analysis.

Examples include:

* Urban Heat Island
* Flood Risk
* Walkability
* Solar Potential
* Noise Analysis
* Urban Morphology

Each plugin declares:

* Required datasets
* Optional datasets
* Processing workflow
* Configuration schema
* Outputs
* Visualization requirements
* Export formats

Plugins never manage execution.

Execution is entirely controlled by the Engine.

---

## Workflow

A Workflow is an executable plan generated by a plugin.

A workflow describes:

* Required tasks
* Task dependencies
* Execution order
* Expected outputs

Workflows remain declarative.

They describe **what** must happen rather than **how** the Engine executes it.

---

## Task

A Task is the smallest meaningful unit of execution.

Examples include:

* Download Sentinel imagery
* Download OpenStreetMap data
* Cloud masking
* Calculate NDVI
* Calculate LST
* Generate heat map
* Export GeoTIFF

Each task contains:

* Inputs
* Outputs
* Dependencies
* Status
* Execution metadata
* Logs

Tasks should be independently executable whenever their dependencies have been satisfied.

---

# Execution Services

The Engine provides shared services used by every Job.

## Plugin Manager

Responsible for:

* Discovering plugins
* Loading plugins
* Validating plugin compatibility
* Registering available analyses

---

## Workflow Manager

Responsible for:

* Building workflows
* Validating workflows
* Resolving task dependencies
* Preparing execution plans

---

## Scheduler

Responsible for:

* Determining execution order
* Resolving dependencies
* Supporting future parallel execution
* Managing execution queues

---

## Dataset Manager

Responsible for:

* Resolving dataset requirements
* Acquiring datasets
* Managing dataset sources
* Preventing duplicate downloads
* Integrating with cache

---

## Cache Manager

Responsible for:

* Storing intermediate results
* Reusing previously computed outputs
* Reducing unnecessary computation
* Supporting resumable execution

---

## Logging Service

Responsible for:

* Recording execution events
* Recording warnings
* Recording errors
* Recording execution timing
* Supporting debugging

Each Job maintains its own execution log while using the Engine's logging infrastructure.

---

## Configuration Manager

Responsible for:

* Loading configuration files
* Validating configuration
* Providing default values
* Resolving runtime parameters

---

## Export Manager

Responsible for:

* Collecting plugin outputs
* Preparing exports
* Delegating visualization
* Generating deliverables

# Execution Lifecycle

Every analysis follows the same lifecycle.

```
User Request
      │
      ▼
Create Job
      │
      ▼
Load Plugin
      │
      ▼
Validate Configuration
      │
      ▼
Generate Workflow
      │
      ▼
Resolve Dataset Requirements
      │
      ▼
Acquire Data
      │
      ▼
Preprocess Data
      │
      ▼
Execute Tasks
      │
      ▼
Generate Outputs
      │
      ▼
Visualization
      │
      ▼
Export
      │
      ▼
Complete
```

The internal algorithms differ between plugins, but the execution lifecycle remains identical.

---

# Job Lifecycle

Every Job progresses through defined states.

```
Created

↓

Validating

↓

Preparing

↓

Running

↓

Completed
```

Possible alternative states include:

* Waiting
* Paused
* Cancelled
* Failed
* Retrying

State transitions should be deterministic and fully logged.

---

# Task Lifecycle

Every task progresses independently through execution states.

```
Pending

↓

Ready

↓

Running

↓

Completed
```

Alternative paths include:

```
Running

↓

Failed

↓

Retrying

↓

Completed
```

or

```
Running

↓

Cancelled
```

Task states are tracked by the Job.

---

# Data Flow

TORANA uses explicit data flow.

Every task must clearly define:

* Required inputs
* Produced outputs

No task should rely on hidden global state.

This improves reproducibility, debugging, and testability.

---

# Dependency Management

Tasks may depend on one another.

Example:

```
Download Sentinel

↓

Cloud Mask

↓

Calculate NDVI

↓

Calculate LST

↓

Normalize

↓

Generate Heat Map
```

The Scheduler executes tasks only after all dependencies have been satisfied.

Independent tasks may execute in parallel in future versions.

---

# Caching Strategy

The Engine supports caching to improve performance.

Cache may include:

* Downloaded datasets
* Intermediate rasters
* Processed vectors
* Temporary calculations

The Cache Manager determines whether previous results can be reused.

Plugins remain unaware of cache implementation.

---

# Logging

Every execution event should be recorded.

Example events include:

* Job creation
* Plugin loading
* Dataset acquisition
* Task execution
* Warnings
* Errors
* Export completion

Logs provide complete traceability for every Job.

---

# Error Handling

The Engine should never terminate unexpectedly because of a recoverable failure.

Possible recovery strategies include:

* Retry failed tasks
* Resume interrupted execution
* Skip optional tasks
* Restore cached results
* Generate informative error reports

Failures should remain isolated to the affected Job whenever possible.

---

# Configuration-Driven Execution

Execution behavior should be controlled through configuration rather than hardcoded values.

Configuration may define:

* Study area
* Analysis type
* Spatial resolution
* Temporal range
* Dataset sources
* Output formats
* Plugin parameters

This enables reproducibility and simplifies sharing analyses.

---

# Engine Guarantees

Every execution performed by TORANA should provide:

* Reproducible execution
* Deterministic workflows
* Explicit inputs and outputs
* Full execution history
* Dataset traceability
* Recoverable failures
* Plugin independence
* Configuration-driven behavior

---

# Future Scalability

The Engine is designed to support future capabilities, including:

* Parallel task execution
* Distributed computation
* Remote execution
* Cloud processing
* Workflow monitoring dashboards
* Job history
* Workflow versioning
* Plugin marketplace
* API-based execution
* Web-based interfaces

These features should be achievable without redesigning the core architecture.

---

# Summary

The TORANA Engine is a generic execution framework for geospatial computation.

It does not implement spatial analyses directly.

Instead, it coordinates plugins through Jobs, Workflows, and Tasks while providing shared execution services such as scheduling, caching, logging, configuration management, and export.

By separating execution from domain-specific logic, the Engine remains stable, extensible, and capable of supporting an expanding ecosystem of geospatial analysis plugins.
